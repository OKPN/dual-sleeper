# Dual Sleeper (デュアル・スリーパー)

**Dual Sleeper** は、Windows PC（特にAI学習サーバーやリモートPC）に最適な、知的で段階的な省電力（電源・モニター管理）システムです。

単なる「時間経過によるスリープ」ではなく、**「ユーザーの物理的な操作」**、**「ネットワークの通信速度」**、および**「GPU（ビデオカード）の動作プロセス」**の3つの軸をインテリジェントに監視し、状況に応じた最適な省電力を自動で実行します。

---

## 💡 具体的なユースケースと動作

本アプリは、日常や開発における以下のような複雑な状況を**「通信量」「GPU使用プロセス」「物理操作」の条件と状態遷移でインテリジェントにハンドリング**し、的確にモニターを消灯してPCをスリープさせます。

1. **ただ放置しているとき（通常の離席）**
   * 👉 操作がなく、通信も発生しないため、最短（無操作2分）で画面を消灯し、その5分後にPC本体をスタンバイ（スリープ）にします。
2. **YouTubeなどの動画サイトを見ているとき（視聴中）**
   * 👉 物理操作はなくても、動画再生による一時的な通信（バッファリング）を検知し、PC本体がスリープするのを防ぎます（動画再生中の快適な動作を維持します）。
3. **大容量ファイルをダウンロードしているとき（放置ダウンロード）**
   * 👉 操作がないためモニターは電気代節約のために先行して消灯（Downloadsフォルダ内の一時ファイル検知により、大容量通信があっても強制消灯します）しつつ、ダウンロード通信が発生している間は、PC本体はスリープせずに稼働し続け、完了を待ちます。
4. **ゲームを起動したまま放置しているとき（寝落ち・キャラ放置）**
   * 👉 操作がないためモニターは消灯し、GPUが高負荷であっても「保護対象（python等）ではない（例: `game.exe`）」かつ「通信が配信クラス（5 Mbps未満）ではない」ため、**ゲームの無駄なグラフィック描画を無視して安全にスリープを実行**し、電気代を劇的にカットします。
5. **ゲームをしていて、かつ配信など重いネットワーク送信が重なっているとき（ゲーム配信中）**
   * 👉 操作がないために万が一モニターが消灯しても、配信データ（5 Mbps以上の高トラフィック）を検知し、**ゲーム配信中と判断してPC本体のスリープを確実に阻止**します。
6. **NVIDIA GPUでAI学習（LoRA学習やローカルLLM推論）などをしているとき**
   * 👉 操作がなく通信が低くても、GPUを消費しているプロセス名に `python.exe`（または `lmstudio.exe` 等）が含まれるため、**AI学習/推論中と判定してスリープを自動で阻止**し、重い学習タスクを最後まで安全に完遂させます。

## 🌟 主な機能

### 1. 段階的な省電力（二段階スリープ）
*   **第一段階（モニターのみ消灯）:** 
    操作がなく、かつ通信量も少ない状態になると、まずはモニターの電源だけを切ります。PC本体は起動したままであるため、バックグラウンド処理やリモートサーバーとしての機能は維持されます。
*   **第二段階（システムをスリープ）:** 
    モニター消灯後、さらに低通信状態が一定時間続いた場合、PC本体をスタンバイ（スリープ）に移行させます。

### 2. 【高度な判定】AI学習とゲーム放置の自動判別（GPU保護機能）
*   **LoRA学習中（スリープ阻止）:**
    NVIDIA製GPUの負荷を監視します。現在GPUを使用しているプロセス名に `python.exe`（PyTorchなどの実行環境）が含まれる場合は、AI学習中とみなして**スリープを自動で阻止**します（モニターの消灯のみ行われます）。
*   **ゲーム放置中（スリープ実行）:**
    メニュー画面などでゲーム（`eldenring.exe` 等）がGPUを高負荷のまま無駄に回し続けている場合は、保護対象外とみなして**安全にスリープを実行**し、無駄な電気代をカットします。
*   ※**NVIDIA製GPU以外の環境（AMD Radeon、Intel/AMD内蔵GPU等）における動作:**
    システム内に `nvidia-smi` が存在しない環境では、プログラムはクラッシュすることなく自動的にこのGPU保護機能をスキップし、「操作 ＋ 通信監視」の高性能スリープツールとして安全に機能し続けます。

### 3. 【ダウンロード自動検知】動画とファイルのスマートな画面消灯制御
YouTubeなどの動画視聴時と、ブラウザによる大容量ファイルのダウンロードは、通信量だけでは区別できません。本アプリはこれをスマートに解決します。
*   **ダウンロード一時ファイルの自動監視:**
    Windows API（`SHGetKnownFolderPath`）経由で、ユーザーの環境に応じた「ダウンロード」フォルダの絶対パス（Dドライブ等に移動されていても対応）を自動取得し、ブラウザの一時ファイル（`.crdownload` または `.part`）を監視します。
*   **動画視聴（画面維持）とダウンロード（画面消灯）の自動切り替え:**
    *   **YouTube動画を視聴中:** 一時ファイルが作られないため、動画のバッファ通信によって画面の自動消灯を防ぎ、快適な視聴を維持します。
    *   **ファイルダウンロード中:** 一時ファイルを検知し、大容量通信があっても**モニター消灯（State 2）への移行を普通に通します。** 一方、PC本体のスリープはダウンロードが完了するまで確実にブロックされ、ダウンロード完了後に初めて自動スリープに入ります。

### 4. 【強固な設計】スリープ失敗時の30秒自動リトライ ＆ 本物復帰判定
Windows Updateの実行中や、特定のアプリがスリープを妨害している場合、OS側でスリープコマンドが一時的に拒否されることがあります。
*   **アグレッシブなスリープとWindows最適化への配慮:**
    本アプリは最短（無操作2分＋5分）でアグレッシブにスリープへ移行させるため、Windowsが裏で行う自動メンテナンスやシステム最適化（デフラグ、セキュリティスキャン等）の時間が減るという側面があります。
    しかし、**Windows Updateや重要な更新処理がバックグラウンドで実際に動いている間は、OS側の安全ガードによってスリープが自動的に拒否（ブロック）されます。**
*   **30秒無限リトライによるスマートな完了待機:**
    一度スリープが決定すると、実際にシステムが眠る（OSのスリープ禁止が解除される）まで30秒おきにスリープコマンドを粘り強く送信し続けます。
    これにより、**「Windows Updateの更新作業を一切邪魔せず正常に進行させ、すべての処理が完了した瞬間の最初の30秒で、PCを最速で自動スリープさせる」**という、極めて合理的で安全な待機動作を実現しています。
*   **10分継続時の自動警告アラート:**
    スリープコマンドが拒否され続け、30秒おきのリトライが連続して**10分以上**経過した場合、スマホへ一度だけ警告（⚠️）を通知し、Windows Updateや他の常駐アプリ（DontSleepなど）に邪魔されている可能性を指摘します。
*   **復帰判定（15秒ルール）と無限スリープ防止:**
    スリープコマンドの前後で「実際の経過時間」を測定します。
    *   **経過時間が15秒未満（即復帰）:** スリープ失敗、またはノイズによる即時誤復帰とみなし、30秒後に再試行します。
    *   **経過時間が15秒以上（本物の復帰）:** 正常にスリープしていたと判断し、タイマーをリセットして「モニター消灯（State 2・再度300秒待ち）」から安全に再開します。これにより、起きた瞬間にまたすぐ眠らされる「無限スリープ地換」を100%回避します。

### 5. 遠隔起動（WOL）に最適な「画面消灯のまま復帰」仕様
*   スリープから復帰（起動）した瞬間、PCは「画面消灯状態（State 2）」のまま目覚めます。外出先からのWOL（Wake-on-LAN）での遠隔起動時に、**誰もいない部屋でディスプレイが勝手に点灯するのを防ぎます。**
*   PCの前にいる人が物理的にマウスやキーボードを触れば、1秒以内に自動で画面が点灯して通常稼働（State 0）へ復帰するため、ローカルでの使用時も違和感なく動作します。

### 6. 時間帯別のスリープモード切り替え
*   **夜間（デフォルト無効、時間指定で有効化）:** 
    長時間使わない時間帯のため、待機電力が完全にゼロになる **「休止状態（ハイバネート）」** に移行します。
*   **それ以外の時間帯:** 
    すぐに作業に復帰できるよう、復帰が高速な **「スタンバイ（スリープ）」** に移行します。
*   ※デフォルト設定（両方 `0`）では、休止状態を非搭載のPCに配慮して「24時間いつでも通常のスタンバイ（スリープ）」になるように安全設計されています。

### 7. Discord ＆ Telegram による「確定実績ベース」のプッシュ通知
*   **スリープ予告（🔔）:** PCが実際にスリープまたは休止状態に入る30秒前にスマホへ通知します。この30秒間の猶予時間中に操作を検知した場合は自動でキャンセルされます。
*   **スリープ完了＆復帰通知（🟢）:** スリープ開始前の「成否が不確定な状態」での完了通知は送信しません。PCが本当にスリープに成功し、**その後起動した（目覚めた）瞬間に、「スリープ開始時刻・解除時刻・実際の睡眠時間」をスマホへ確定実績として事後報告**します。
*   **リトライ監視の静音化:** スリープが拒否された場合、初回の失敗時のみ警告（⚠️）を通知し、リトライ中は一切通知を送らず無音化することで、チャットが通知で汚れるのを100%防止します。

## 💡 Wake on LAN (WoL) との最強の組み合わせ推奨

**Dual Sleeper** は、**WoL (Wake on LAN)** 技術と組み合わせることで、電気代を極限まで節約した完璧な「プライベート・リモート開発サーバー / リモートPC」を構築することができます。

### なぜ組み合わせるべきなのか？ (相乗効果)
1. **必要なときだけ遠隔起動 (WoL):**
   外出先やベッドの中からスマホのWoLアプリ（またはルーター）を使って、スリープ中のPCにパケットを送信し、必要なときだけPCをリモート起動できます。
2. **用が済んだら自動で勝手に眠る (Dual Sleeper):**
   リモート操作を終えて接続を切断するか、起動したまま寝落ちして放置した場合、Dual Sleeperが「無操作状態」かつ「バックグラウンドのAI学習やダウンロードが完了したこと」を自動検知し、安全にスリープ（または休止状態）へと戻してくれます。
3. **点灯ノイズの防止:**
   Dual Sleeperは「画面消灯のまま復帰」する仕様になっているため、WoLで遠隔起動した際に、誰もいない暗い部屋でディスプレイが勝手に眩しく光り出すこともありません。

このように組み合わせることで、**「外出先から安全に起動し、用事が済んだら（または寝落ちしたら）勝手に自動スリープに戻る」という、電気代がほぼゼロに近い理想的なリモートデスクトップ/AI学習環境**が完成します。ぜひルーターやマザーボードのWoL機能と組み合わせてご活用ください。

## 📂 ファイル構成

*   `dual_sleeper.py`: メインのPythonプログラム。
*   `test_sleep_block_monitor.py`: **[新機能]** OSのスリープを妨げている犯人を特定するリアルタイムチェッカー。
*   `test_sleep_block_monitor.bat`: **[新機能]** ダブルクリック（「管理者として実行」推奨）でチェッカーを起動するバッチファイル。
*   `config.json.example`: 設定ファイルのひな形（通知等の秘匿情報を除外したもの）。
*   `config.json`: **[ローカル専用]** 実際の動作設定ファイル（`.gitignore` によりGit管理から自動除外されます）。
*   `requirements.txt`: 動作に必要な外部ライブラリ（`psutil`）を指定。
*   `run.bat`: ライブラリの準備からスクリプトの起動までをダブルクリックで行うバッチファイル。
*   `test_webhook.bat`: **[テスト用]** `config.json` に設定したDiscord / Telegram通知が正しく動くか検証するバッチファイル。

---

## 🚀 使い方

### 1. 設定ファイルの作成（初回のみ）
1.  このフォルダ内にある `config.json.example` をコピーします。
2.  コピーしたファイルを **`config.json`** にリネームします。
3.  `config.json` をメモ帳などで開き、各種しきい値や通知用の設定を行います。

### 2. 起動方法
1.  **`run.bat`** をダブルクリックして実行します。
2.  自動的に必要なライブラリ（`psutil`）がインストールされ、タイムスタンプ付きの監視コンソールが立ち上がります。

## ⚙️ 設定オプション (`config.json`)

| 設定項目 | 初期値 | 単位 | 説明 |
| :--- | :--- | :--- | :--- |
| `idle_limit_seconds` | `120` | 秒 | モニター消灯を検討し始める無操作時間（2分）。 |
| `network_limit_kbs` | `20.0` | KB/s | 通常の無通信状態とみなす通信速度（送受信合算）のしきい値。 |
| `network_check_duration_seconds` | `60` | 秒 | 通信量がしきい値以下であるかを確認し続ける判定時間（1分）。 |
| `check_interval_seconds` | `5` | 秒 | PCの稼働状態をチェックする周期。 |
| `standby_after_monitor_off_seconds` | `300` | 秒 | モニターが消灯したあと、PC本体をスリープさせるまでの待機時間（5分）。 |
| `hibernate_start_hour` | `0` | 時 | 夜間休止状態（ハイバネート）を適用する開始時刻（0で無効）。 |
| `hibernate_end_hour` | `0` | 時 | 夜間休止状態（ハイバネート）を適用する終了時刻（0で無効）。 |
| `force_monitor_off_idle_seconds` | `900` | 秒 | 通信の有無に関わらず、無操作がこの秒数続いた場合にモニターを強制オフにする時間（15分）。 |
| `discord_webhook_url` | `""` | URL | DiscordのWebhook URL。空のままだと通知は無効になります。 |
| `telegram_bot_token` | `""` | トークン | Telegram BotのAPI Token（空で無効）。 |
| `telegram_chat_id` | `""` | ID | Telegramの宛先チャットID（空で無効）。 |
| `sleep_pending_seconds` | `30` | 秒 | スリープ予告を通知してから、実際にスリープするまでの猶予時間。 |
| `gpu_protect_processes` | `["python.exe", "python"]` | リスト | スリープを阻止したい現在GPUを使用しているプロセス名のリスト。（※LM Studio等を保護したい場合は、ここに `"lmstudio.exe"` 等を追加してください） |
| `gpu_limit_percent` | `10` | ％ | GPU使用率がこの％以上で、かつ上記保護対象プロセスがGPUを使用している場合にスリープを阻止します。 |
| `high_network_limit_kbs` | `625.0` | KB/s | ゲーム配信中などの高トラフィックとみなすしきい値（5 Mbps相当）。この速度以上の通信がある場合は、いかなる場合もスリープを阻止します。 |

## 🔔 Discord ＆ Telegram の通知テスト方法
1.  `config.json` に設定を記述します。
2.  フォルダ内にある **`test_webhook.bat`** をダブルクリックして実行します。
3.  設定したDiscordチャンネル、またはTelegramのチャットにテスト通知が届けば、連携は無事に完了しています！
